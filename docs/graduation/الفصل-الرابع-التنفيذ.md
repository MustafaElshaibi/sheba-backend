## 4.2 تنفيذ الوحدات الرئيسية (Module-by-Module Implementation)

### 4.2.1 وحدة الهوية (Identity) — مرساة الثقة

أهم وحدة وأكثرها تعقيداً: خادم OpenIddict، دورة حياة الحساب (eKYC → موافقة → نشط)، المشرفون، سجلّ
الأطراف المعتمدة، إصدار/تحقق OTP، ومزوّدات NID/OTP القابلة للاستبدال.

**قرار تصميمي محوري: منحة OAuth مخصّصة بدل منحة كلمة المرور.**
حيث أسقط OAuth 2.1 منحة كلمة المرور تماماً، عُرِّفت منحة امتداد مسمّاة `urn:sheba:grant:national_id_otp`
لتمثيل «كلمة مرور + OTP هاتفي» كطلب رمز من الدرجة الأولى:

```csharp
// OidcEndpoints.cs — التحقق من OTP قبل سكّ أي رمز
private static async Task<IResult> IssueCitizenTokenAsync(OpenIddictRequest request, IMediator mediator)
{
    var accountIdRaw = request.GetParameter("account_id")?.Value?.ToString();
    var otp = request.GetParameter("otp")?.Value?.ToString();

    if (!Guid.TryParse(accountIdRaw, out var accountId) || string.IsNullOrWhiteSpace(otp))
        return TokenError(Errors.InvalidRequest, "account_id and otp are required for this grant.");

    var result = await mediator.Send(new VerifyLoginOtpCommand(accountId, otp));
    if (result.IsFailure)
        return TokenError(Errors.InvalidGrant, result.Error!.Message);

    // ... بناء المطالبات (sub, name, email, national_id_hash, loa, role=Citizen)
    // civil_data لا تُمنح تلقائياً — يجب طلبها صراحةً وتتطلّب LoA >= 2
}
```

**قرار تصميمي: مولّد OTP في طبقة التطبيق لا في المزوّد (T-SEC-8).**
التوليد والتجزئة وسياسة المحاولات تعيش في `Shared.Kernel`؛ المزوّد (Twilio/Console) يستطيع فقط
**تسليم** رمز يُسلَّم إليه، لا توليده أو إضعافه.

### 4.2.2 وحدة الوزارات (Ministry) — العمود الفقري للتكامل

سجلّ + خزنة اعتماد. الفصل بين التهيئة والاعتماد يتيح سرد التهيئات دون تجسيد الأسرار. الاعتمادات تُفكّ
فقط داخل مُحوّلات المصادقة وقت الاستدعاء.

**قرار تصميمي: تشفير AES-256-GCM** (nonce 12 بايت ‖ نص مشفّر ‖ tag 16 بايت، base64) بدل RSA-OAEP
القديم الذي كان خطأً تصميمياً (أداة لفّ مفاتيح لا تشفير حقول). مُحوّلات المصادقة الخمسة (OIDC، OAuth2،
API Key، Bearer، Basic) مبنية بنمط **Strategy**.

### 4.2.3 وحدة طلبات الخدمة (ServiceRequest) — محرك سير العمل

كتالوج هرمي (فئة → خدمة → مخطط نموذج، رسوم، مستندات، خطوات) + دورة حياة مدفوعة بمحرك خطوات
(`IWorkflowStepHandler` لكل `WorkflowStepType`). خمسة معالجات خطوات: Payment، MinistryCall،
WebhookWait، AdminReview، AutoComplete.

### 4.2.4 وحدة المستندات (Document)

بيانات وصفية + تخزين MinIO؛ روابط تنزيل مُوقّعة مسبقاً صالحة 15 دقيقة؛ حذف ناعم. الملكية تأتي من
`sub` في الرمز، والمعالج يعيد `null` لغير المالك حتى لا يتسرّب وجود المستند.

### 4.2.5 وحدة المحفظة (Wallet)

سجلّ DID، إصدار VC كـ JWT-VC موقّع (RSA JWS وفق نموذج W3C). عند الموافقة على الهوية، تُصدَر تلقائياً
شهادة `DigitalIdentityCredential`.

### 4.2.6 الوحدات المساندة
- **Payment:** أوامر دفع وهمية مع مقبس `IPaymentGateway` قابل للاستبدال.
- **Notification:** استهلاك أحداث وتسليم بريد/SMS + سجلّ (لا واجهة API عامة — مدفوعة بالأحداث).
- **Audit:** سجلّ إلحاقي فقط يُكتب بسلوك خط أنابيب MediatR لكل أمر مُغيِّر للحالة.
- **Admin:** نموذج قراءة BI + تقارير (PDF/Excel/CSV) + Hangfire.
- **Citizen:** ملف تعريف موسّع (لا نقاط نهاية عامة حالياً — يُنشَأ عبر حدث الموافقة).

## 4.3 مقتطفات كود للأنماط المتقاطعة (Cross-cutting Patterns)

### نمط «الهوية من الرمز لا من الجسم» (Actor identity from token)
منع ثغرة انتحال الهوية: معرّف الفاعل يُؤخذ دائماً من `sub` في الرمز الموثّق، لا من جسم الطلب.

```csharp
// IdentityModule.cs — الموافقة على طلب هوية
admin.MapPost("/{requestId:guid}/approve", async (Guid requestId,
    ApproveIdentityRequestBody body, ClaimsPrincipal user, IMediator mediator, CancellationToken ct) =>
{
    // هوية المراجع من رمزه هو (sub)، لا من حقل في الجسم —
    // وإلا لأمكن لأي مراجع نسب قراره إلى مشرف آخر.
    var result = await mediator.Send(
        new ApproveIdentityRequestCommand(requestId, user.RequireSubjectId(), body.Notes), ct);
    return result.ToHttpResult();
});
```

### مظروف JSend موحّد عبر مرشّح مشترك (Shared JSend filter)
لا يُكتب أي مظروف استجابة يدوياً في نقاط النهاية؛ `JSendWrappingFilter` يلفّ كل استجابة (عدا `/connect/*`
و`/.well-known/*`).

### الأحداث الموثوقة عبر Transactional Outbox
`OutboxSaveChangesInterceptor` يحوّل الأحداث المرفوعة على الكيانات إلى صفوف `outbox_messages` في **نفس**
استدعاء `SaveChanges` مع كتابة الكيان، فلا يضيع حدث ولا يُرسَل حدث لمعاملة مُتراجَع عنها.

### منع تعداد الحسابات (Anti-enumeration, BR-ON-3)
أي فشل في التحقق (غير موجود، هاتف غير مطابق، متوفّى، موقوف...) يعيد **رسالة خطأ عامة موحّدة** — لا
كاشف تعداد، تماماً كما تفعل GOV.UK One Login وUAE Pass.

## 4.4 التحديات وحلولها (Challenges & Solutions)

هذه تحدّيات حقيقية واجهها المشروع وعولجت (موثّقة في `docs/sheba.md` و`known-issues.md`):

| # | التحدي | الجذر التقني | الحل |
|---|--------|--------------|------|
| 1 | **تسرّب مطالبة `oi_prst`** في جسر الجلسة | نسخ مطالبات الرمز حرفياً إلى كوكي الجلسة نقل مطالبات OpenIddict الداخلية، فرفض التبادل بـ «issued to a different client» لكل RP خارجي | ترشيح المطالبات الخاصّة (`oi_*` + `aud/exp/iat/iss/jti/nbf/scope/client_id`) قبل النسخ |
| 2 | **فشل كشف سرقة الرمز صامتاً** | رسالة سبب الإلغاء (57 حرفاً) تجاوزت عمود `varchar(50)` فرمت `DbUpdateException` بدل الإلغاء — أي فشل في الاتجاه الذي يترك العائلة **غير مُلغاة** | تقصير الرسالة + اقتطاع دفاعي في `RefreshTokenFamily.Revoke()` |
| 3 | **`ValidationBehavior` صامت (خطأ 500)** | مسح مجمّعات Infrastructure فقط لم يسجّل أي مُتحقّق (المُتحقّقات في طبقة Application) | مسح مجمّعات Application **و**Infrastructure معاً في `Program.cs` |
| 4 | **غياب فرع منحة `authorization_code`** | كان `/connect/token` يدعم المنحتين المخصّصتين + refresh فقط، فلا شيء يستبدل الرمز الذي تنتجه سباكة PKCE | إضافة فرع `IssueFromAuthorizationCodeAsync` — اكتُشف بالتحقق الحيّ لا بالاختبار الوحدوي |
| 5 | **ثغرة الهوية المُورَّدة من العميل** | كانت أجسام الطلبات تحمل `citizenId`/`accountId` مباشرة، فأمكن التصرّف باسم غيرك | أخذ الهوية دائماً من `sub` في الرمز (نمط §4.3) عبر LoA upgrade، تقديم الطلب، ملكية المستند |
| 6 | **صمود أعطال الوزارات** | استدعاءات HTTP خارجية بطيئة/فاشلة تُعلّق الطلب | خط مرونة Polly (Retry + Circuit-Breaker + Timeout) + خطوة `WebhookWait` غير متزامنة |
| 7 | **عزل السجل المدني المجهول** | شكل واجهة السجل الحقيقي غير معروف | منفذ `INationalIdProvider` + مزوّد وهمي بـ 8 مواطنين + مزوّد HTTP مستقبلي خلف نفس العقد |

> كل تحدٍّ من 1، 2، 4 اكتُشف أثناء **التحقق الحيّ من طرف إلى طرف** لتدفّق OIDC، لا في الاختبارات
> الوحدوية — ما يبرز أهمية اختبار التكامل الحقيقي.

أمنية مدفوعة بالاكتشاف الحيّ، تعكس عمق فهم المشكلة لا مجرّد إتمامها.
